# v0.2 — Library & Import

Bringing the offline taste-data flywheel into the app. This is the next direction after the v0.1 cleanup. Read `AGENTS.md` before working on any item here.

---

## Why this is the next direction

Without `data/taste/tracks.evidence.json` the LLM planner runs on shallow prompt signals, and the product is "a generic radio that happens to have a host." The thing that makes a private AI radio feel like **yours** is the loop:

```
import → analyze → see what it learned → use it → it gets sharper
```

Today that loop is CLI-only (`scripts/analyze-playlist-openai.mjs`, `scripts/fetch-netease-lyrics.mjs`, `scripts/build-taste-profile.mjs`). Most users can't run it. Putting it in the app is what turns "demo of a music player" into "my personal AI station."

Other candidate directions (scheduler, PWA shell, WebSocket realtime, multi-provider, audio polish) are real but lower-leverage. They sit on top of taste data, not next to it.

## Architectural decisions

### 1. Import and analyze are two distinct operations

Import is cheap, deterministic, no LLM calls. Analyze is expensive, slow, requires an API key. Bundling them would burn user tokens on accidental clicks.

| | Import | Analyze |
|---|---|---|
| Cost | free | per-track LLM calls |
| Duration | seconds | minutes |
| Determinism | yes | partly (LLM variance) |
| Idempotent | yes | re-runs overwrite per-track files |
| API key needed | no (mock or real) | yes (`LLM_API_KEY`) |

UI shows two buttons. The analyze button always displays an estimated call count before triggering.

### 2. Lyrics fetching is part of import, not analyze

Lyrics fetching is deterministic and free, and lyrics dramatically improve analysis confidence. We chain it into the import step so by the time a user clicks "Analyze," the lyrics are already on disk.

CLI script `scripts/fetch-netease-lyrics.mjs` is the existing logic — folded into the import endpoint.

### 3. Per-track evidence files; `tracks.evidence.json` is derived

**Current (v0.1) state:**
- `data/taste/tracks.evidence.json` is the single source.
- Re-running analysis on a new playlist **overwrites** it.

**New (v0.2) layout:**

```
data/taste/
  tracks/
    netease/
      123456.json   # one file per track
      789012.json
  tracks.evidence.json   # DERIVED — regenerated by runtime + scripts/build-evidence-aggregate.mjs
  profile.md             # DERIVED — regenerated by scripts/build-evidence-aggregate.mjs only (offline)
  rules.json             # DERIVED — regenerated by scripts/build-evidence-aggregate.mjs only (offline)
```

The runtime job only rewrites `tracks.evidence.json` after a successful analysis run. `profile.md`
and `rules.json` are larger downstream artifacts whose generators live in
`scripts/build-taste-profile.mjs` — they stay stale until the offline script is re-run. Porting
that generator into the Kotlin runtime is a v0.3 task; until then the planner reads the most recent
profile/rules from disk, which is whatever the script last produced.

Each per-track file is the same shape as one entry of the old `tracks.evidence.json.tracks[]` array. Adding a playlist appends new files; analyzing the same track again rewrites just that file.

`tracks.evidence.json` becomes a build artifact: `scripts/build-evidence-aggregate.mjs` (new, small) walks `tracks/**/*.json` and writes the aggregate. Runtime `TasteProfileRepository` keeps reading `tracks.evidence.json` as today — no runtime change.

**Conflict rule** (when does a value get overwritten):
1. Manual override (UI-edited, v0.3) always wins.
2. Newer LLM analysis beats older.
3. Lyrics-backed evidence beats metadata-only.

For v0.2 (no UI overrides yet), rule 2 + 3 are enough.

### 4. Job model for long-running analysis

Analysis can take minutes. HTTP requests don't wait. Use a job pattern:

- `POST /api/import/.../analyze` returns immediately with `{jobId, estimatedCalls, estimatedCostUsd}`.
- `GET /api/jobs/{jobId}` returns `{status, processed, total, current, errors[], startedAt, finishedAt}`.
- `DELETE /api/jobs/{jobId}` cancels (in-flight track finishes, queue drains).
- Per-track failures: log `needsReview: true` on that file, continue. Partial evidence beats no evidence.

Frontend polls `GET /api/jobs/{jobId}` every 1–2s. WebSocket is a v0.3 upgrade if polling becomes a real pain point.

### 5. Settings page never echoes API key values

The Settings view shows only **status** per integration:

- `LLM_API_KEY`: ✓ configured / ✗ not set
- `FISH_API_KEY`: ✓ configured / ✗ not set
- `NETEASE_COOKIE`: ✓ configured / ✗ not set
- `OPENWEATHER_API_KEY`: ✓ configured / ✗ not set

Never display the value. Never let the value round-trip through the UI.

## API surface

New / changed endpoints. All routes are under `/api`.

### Imports

```
POST /api/import/playlist
  body:    { "source": "https://music.163.com/#/playlist?id=123456" }
  returns: { slug, playlistId, name, trackCount, importedAt, lyricsFetched }
  note:    Extended from v0.1 — now also fetches lyrics inline.

GET /api/imports
  returns: ImportRecord[]
           { slug, name, trackCount, importedAt, analyzedAt | null, status }

GET /api/imports/{slug}
  returns: { ...ImportRecord, tracks: TrackSummary[] }
```

### Analysis jobs

```
POST /api/imports/{slug}/analyze
  body:    { "force": false, "trackIds": [] | null }
  returns: { jobId, estimatedCalls, estimatedCostUsd, model }
  note:    force=true re-analyzes tracks that already have evidence.
           trackIds=null analyzes everything; else just those.

GET /api/jobs/{jobId}
  returns: { jobId, status: "running" | "completed" | "cancelled" | "failed",
             processed, total, current: TrackSummary | null,
             errors: [{ trackId, message }],
             startedAt, finishedAt | null }

DELETE /api/jobs/{jobId}
  returns: { jobId, status: "cancelled" }
```

### Taste / Library

```
GET /api/taste/tracks
  query:   ?language=&minConfidence=&tag=&sort=&limit=&offset=
  returns: { tracks: TaggedTrackView[], total }

GET /api/taste/tracks/{provider}/{id}
  returns: EvidenceTrackAnalysis  (the existing model from Models.kt)

GET /api/taste/profile
  returns: { profileText, rules, source }
```

`TaggedTrackView` is a UI-shaped subset (smaller than `EvidenceTrackAnalysis`): id, title, artist, album, coverUrl, language, dominantTags (≤5), scores (energy/valence/night/coding/skipRisk), needsReview, lastAnalyzedAt.

### Notes

- API contract types live in `Models.kt` (Kotlin) and **must** be mirrored in `apps/web/src/types.ts`. See AGENTS.md §5 — API contracts must match exactly.
- Job state is in-memory for v0.2. Persistence (resuming after server restart) is a v0.3 concern.

## Frontend information architecture

Four views, simple left-nav switching. No router library required if `useState<View>` keeps it simple.

```
┌──────────────────────────────────────────────────────┐
│  Aftertaste FM                                       │
├──────┬───────────────────────────────────────────────┤
│ ▶    │  Player        existing main view             │
│ ♫    │  Library       see analyzed tracks            │
│ ↓    │  Import        import + analyze + history     │
│ ⚙    │  Settings      host + weather + integrations  │
└──────┴───────────────────────────────────────────────┘
```

### Player (existing)

No structural change. Keep the current Agent + Playback + Queue layout. Possibly trim the bottom status strip once Settings becomes its own page.

### Library

- Sortable / filterable table of tracks.
- Columns: cover · title · artist · language · energy · night · coding · confidence · `needsReview` flag.
- Click a row → right-side detail panel.
- Detail panel: full `EvidenceTrackAnalysis` with tag chips, score bars, evidence sources (`metadata` / `lyrics` / `model_inference`), lyric excerpt.
- Read-only. Override moves to v0.3.

### Import

- URL/id input + "Import" button.
- Below: list of past imports (`/api/imports`).
- Each row: name, track count, imported time, "Analyze" button (with call/cost estimate), progress bar when analyzing, cancel button.
- Failed tracks shown collapsed under the row with reason.

### Settings

- Host configuration: language, style, name.
- Weather location.
- Provider selector (`mock` / `netease`).
- Integration status list (LLM / Fish / Netease cookie / OpenWeather) — status only, never values.

## Pitfalls (already-shipped or already-tempting mistakes — do not repeat)

These match AGENTS.md anti-patterns and add a few specific to v0.2.

- **Do not bundle import and analyze into one button.** Token cost will surprise users.
- **Do not run analysis synchronously on the request thread.** Use the job model.
- **Do not fail the whole analysis when one track fails.** Mark that track `needsReview: true` and continue.
- **Do not silently overwrite analysis on re-import.** Per-track files + explicit `force=true` flag.
- **Do not load the entire library client-side without pagination.** Use `limit/offset` on `/api/taste/tracks`.
- **Do not display API key values in Settings.** Status only.
- **Do not use a magic number `3` or `30` anywhere new.** Named constants per AGENTS.md §3.
- **Do not add a fallback path to mock data when analysis fails.** Per AGENTS.md §1 — surface the error.
- **Do not start growing keyword lists in the new views.** Filter UI is structured (language picker, tag select), not substring search in React. Per AGENTS.md §4.
- **Do not let `App.tsx` exceed 300 lines again.** Each view is its own file in `components/views/`. Per AGENTS.md §2.
- **Do not stringify a typed track filter into a query param the server re-parses with regex.** Pass typed query params (`?language=en&minConfidence=0.6&tag=late-night`); parse them once.

## Definition of done (v0.2)

A working v0.2 means:

1. From the web UI alone, a user can: paste a Netease playlist URL → see it imported → click Analyze → watch progress → see new tracks appear in the Library view → cancel if needed.
2. `data/taste/tracks/<provider>/<id>.json` is the source of truth; `tracks.evidence.json` is regenerated from it on every analysis run.
3. Re-importing the same playlist + clicking Analyze with `force=false` is a no-op (skip already-analyzed).
4. Running analysis on a 100-track playlist returns within ~10s of user clicking "Analyze" (job started; progress visible). Completion takes whatever the model takes; partial results are visible mid-run.
5. Server restart preserves all imported and analyzed data (it's on disk).
6. Server restart cancels in-flight jobs gracefully (no zombie state).
7. All AGENTS.md hard rules still pass: `./gradlew compileKotlin`, `tsc -b`, `tsc --noEmit` green; no file over 300 lines; no new silent `runCatching{}.getOrNull()`; no new substring matching outside `IntentExtractor`.
8. `docs/api.md` documents every new endpoint with example request and response.

If any of 1–7 fails, v0.2 is not done. Do not move on to v0.3 (manual override, multi-provider, scheduler).
